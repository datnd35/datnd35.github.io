# DISC Framework cho Tech Lead — Giao tiếp đúng kiểu người

> Một leader tốt là người biết điều chỉnh cách giao tiếp theo kiểu người đối diện — không phải theo cách mình thích.

---

## 1. DISC là gì?

**DISC** phân loại phong cách hành vi/giao tiếp thành 4 nhóm:

```
D = Dominance      → Người quyết đoán, thích kết quả
I = Influence      → Người cởi mở, thích kết nối
S = Steadiness     → Người ổn định, thích an toàn
C = Conscientious  → Người cẩn thận, thích logic/chính xác
```

DISC không nói ai giỏi hơn ai. Nó giúp bạn hiểu:

```
Người này quan tâm điều gì?
Người này sợ điều gì?
Nên giao tiếp với họ thế nào?
Nên giao task và feedback thế nào?
```

## 2. Diagram tổng quan DISC

```
                      PEOPLE / RELATIONSHIP
                    Quan tâm con người, cảm xúc
                              ▲
                              │
              I               │              S
        Influence              │         Steadiness
  Năng lượng, giao tiếp        │     Ổn định, hỗ trợ
  Thích kết nối, ý tưởng       │     Thích an toàn, rõ ràng
                              │
FAST ◄────────────────────────┼────────────────────────► SLOW
Nhanh, trực diện              │                     Chậm, thận trọng
Thích hành động               │                     Thích chắc chắn
                              │
              D               │              C
        Dominance              │      Conscientious
  Quyết đoán, kết quả          │     Logic, tiêu chuẩn
  Thích kiểm soát              │     Thích dữ liệu, đúng sai
                              │
                              ▼
                        TASK / LOGIC
                   Quan tâm công việc, kết quả
```

## 3. Nhóm D — Dominance

```
D = Direct + Result-oriented
```

### Đặc điểm

```
- Nói thẳng, nhanh, trực tiếp
- Thích kết quả rõ ràng, không thích vòng vo
- Thích kiểm soát tình huống
- Có thể hơi áp lực hoặc thiếu kiên nhẫn
```

### Họ thường hỏi

```
Kết quả là gì? / Deadline khi nào?
Rủi ro lớn nhất là gì?
Có cách nào nhanh hơn không?
Ai chịu trách nhiệm?
```

### Cách giao tiếp

| Nên | Tránh |
|---|---|
| Đi thẳng vào vấn đề | Giải thích quá dài, vòng vo |
| Nói outcome trước, detail sau | Nói cảm xúc quá nhiều |
| Đưa option rõ ràng | Thông tin mơ hồ |

### Ví dụ câu nói

```
I recommend option A because it reduces delivery risk and helps us meet the sprint goal.
The main risk is regression on the user role flow.
I need your confirmation today so the team can continue implementation.
```

---

## 4. Nhóm I — Influence

```
I = Expressive + People-oriented
```

### Đặc điểm

```
- Nói nhiều, thân thiện, dễ tạo không khí
- Thích ý tưởng mới, thích được công nhận
- Có khả năng kết nối team
- Đôi khi thiếu chi tiết hoặc dễ đổi hướng
```

### Họ thường hỏi

```
Team nghĩ sao? / User có thích không?
Có cách nào thú vị hơn không?
Ai sẽ join phần này?
Có thể demo sớm không?
```

### Cách giao tiếp

| Nên | Tránh |
|---|---|
| Tạo không khí tích cực | Quá khô khan, chỉ nói số liệu |
| Công nhận ý tưởng của họ trước | Dập ý tưởng quá nhanh |
| Sau đó kéo về action cụ thể | Bỏ qua cảm xúc của họ |

### Ví dụ câu nói

```
That's a good idea and it could improve the user experience.
To make it practical for this sprint, I suggest we split it into two parts:
first, deliver the basic flow; second, enhance the UI in the next iteration.
```

---

## 5. Nhóm S — Steadiness

```
S = Supportive + Stable
```

### Đặc điểm

```
- Điềm tĩnh, kiên nhẫn, làm việc đáng tin cậy
- Thích môi trường ổn định, không thích xung đột
- Có thể ngại nói ra vấn đề hoặc ngại thay đổi nhanh
```

### Họ thường hỏi

```
Yêu cầu này đã chắc chưa?
Có ảnh hưởng đến phần cũ không?
Mình có đủ thời gian không?
Có tài liệu/hướng dẫn không?
```

### Cách giao tiếp

| Nên | Tránh |
|---|---|
| Nói rõ expectation, từng bước | Ép quyết định quá nhanh |
| Cho họ cảm giác an toàn | Gây áp lực đột ngột |
| Hỏi nhẹ nhàng để họ chia sẻ khó khăn | Thay đổi requirement liên tục không giải thích |

### Ví dụ câu nói

```
This change may affect the existing flow, but we will handle it step by step.
Please start with the UI part first.
If you find any unclear requirement or regression risk, let me know early and we will review it together.
```

---

## 6. Nhóm C — Conscientious

```
C = Careful + Quality-oriented
```

### Đặc điểm

```
- Cẩn thận, logic, quan tâm chất lượng
- Thích dữ liệu, rule, document
- Giỏi phân tích risk
- Có thể hơi chậm vì muốn chắc chắn
```

### Họ thường hỏi

```
Requirement cụ thể là gì? / Acceptance criteria là gì?
Có edge case nào không?
Data source ở đâu? / Có document/spec không?
Cách này có ảnh hưởng maintainability không?
```

### Cách giao tiếp

| Nên | Tránh |
|---|---|
| Chuẩn bị dữ liệu/spec rõ | Nói chung chung |
| Nói logic, evidence, rule | Ép làm khi requirement chưa rõ |
| Cho thời gian phân tích | "Cứ làm đi rồi tính" |

### Ví dụ câu nói

```
The acceptance criteria are clear for the main flow, but there are still two edge cases to confirm.
Please help me review the validation logic and identify possible regression areas before implementation.
```

## 7. Bảng so sánh nhanh

| Nhóm | Mạnh về | Sợ nhất | Khi giao tiếp nên | Tránh |
|---|---|---|---|---|
| **D** | Kết quả, quyết định | Mất kiểm soát, chậm trễ | Ngắn gọn, trực tiếp, outcome rõ | Vòng vo, quá nhiều detail |
| **I** | Kết nối, ý tưởng | Bị bỏ qua, mất tương tác | Tích cực, công nhận ý tưởng | Quá khô khan, phủ định nhanh |
| **S** | Ổn định, hỗ trợ | Thay đổi đột ngột, xung đột | Nhẹ nhàng, rõ từng bước | Gây áp lực mạnh |
| **C** | Logic, chất lượng | Sai sót, thiếu dữ liệu | Có evidence, spec, rule rõ | Mơ hồ, "cứ làm đi" |

## 8. Nhận biết nhanh kiểu người

```
Người này giao tiếp thế nào?
              │
   ┌──────────┴──────────┐
   │                     │
 Nhanh                 Chậm
   │                     │
 ┌─┴──┐              ┌───┴──┐
Task  People       People  Task
 │       │           │       │
 ▼       ▼           ▼       ▼
 D       I           S       C
```

Công thức nhanh:

```
Nhanh + Task   → D
Nhanh + People → I
Chậm + People  → S
Chậm + Task    → C
```

## 9. Giao task theo từng nhóm

### Với nhóm D

```
Task phù hợp: cần quyết định nhanh, ownership rõ, deadline rõ

"This task has a tight deadline.
The expected outcome is to complete the main flow by Wednesday.
You can choose the approach, but please flag any blocker early."
```

### Với nhóm I

```
Task phù hợp: demo, UI/UX, phối hợp nhiều người, communicate với stakeholder

"This feature needs good user experience and team coordination.
Please prepare a quick demo and collect feedback before we finalize."
```

### Với nhóm S

```
Task phù hợp: ổn định, follow process, support/review/regression

"Please follow the current flow and update only the necessary parts.
I will share the existing behavior and test cases.
If anything is unclear, let's clarify together before you start."
```

### Với nhóm C

```
Task phù hợp: logic phức tạp, quality cao, refactor, validation/security

"Please focus on the validation rules and possible edge cases.
Before implementation, help me confirm whether this change affects any existing logic."
```

## 10. Feedback theo từng nhóm

```
FEEDBACK USING DISC
    │
    ├── D: Nói thẳng impact + action cần làm
    ├── I: Công nhận trước → điều chỉnh sau
    ├── S: Nhẹ nhàng + tạo cảm giác an toàn
    └── C: Dựa trên evidence + tiêu chuẩn rõ
```

### Feedback cho D

```
The result is not aligned with the expected outcome.
The main issue is missing role validation.
Please fix this part first because it blocks the release.
```

### Feedback cho I

```
Your idea is good and the UI looks more engaging.
One thing to improve is consistency with the existing design.
Let's adjust the spacing and button behavior to match the current pattern.
```

### Feedback cho S

```
Thanks for handling this carefully.
I noticed one part may affect the existing flow.
Let's review it together and update the test case to avoid regression.
```

### Feedback cho C

```
The implementation is clean, but one edge case is missing.
Based on the acceptance criteria, we also need to handle the empty data scenario.
Please add this case before we merge.
```

## 11. DISC khi họp với khách hàng

### Khách kiểu D

Họ hỏi:

```
Why does it take so long? / Can we reduce the estimate?
What is the risk? / When can you finish?
```

Trả lời:

```
The estimate is high because this task impacts three areas:
UI implementation, role validation, and regression testing.
If we reduce the scope to UI only, we can deliver faster.
If role validation is included, we need additional time to avoid production issues.
```

### Khách kiểu C

Họ hỏi:

```
Can you explain the estimate? / Which part takes the most time?
What are the dependencies? / Do we have test coverage?
```

Trả lời:

```
The estimate includes four parts:
requirement clarification, implementation, unit testing, and regression testing.
The most time-consuming part is regression because this flow is shared by multiple user roles.
```

### Khách kiểu I

Họ hỏi:

```
Can we make this better for users? / Can we show a demo soon?
```

Trả lời:

```
Yes, we can prepare a quick demo for the main flow first.
After that, we can collect feedback and improve the UI in the next iteration.
```

### Khách kiểu S

Họ hỏi:

```
Will this change affect existing behavior? / Does the team have enough support?
```

Trả lời:

```
There is some impact on existing behavior, so we will handle it carefully.
We plan to split the work into implementation and regression testing to reduce risk.
```

## Tóm tắt dễ nhớ

```
D = Get it done      → Tôi cần kết quả.
I = Make it exciting → Tôi cần sự kết nối.
S = Keep it stable   → Tôi cần sự ổn định.
C = Do it correctly  → Tôi cần sự chính xác.
```

> **DISC không dùng để "dán nhãn" người khác.**  
> DISC dùng để hiểu người nhanh hơn, giao tiếp phù hợp hơn, giao task đúng hơn và feedback hiệu quả hơn.
